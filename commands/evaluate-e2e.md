# E2E 스크립트 품질 평가

E2E 테스트 스크립트를 3가지 관점에서 평가하고 개선점을 리포트합니다.

---

## 입력

`$ARGUMENTS` = E2E 테스트 파일 경로 (또는 `all`로 전체 평가)

```bash
# 단일 파일
/evaluate-e2e e2e/purchase-request-detail.spec.ts

# 전체
/evaluate-e2e all
```

---

## 평가 기준 (3가지)

### 1. 스크립트 품질 (Script Quality) — 40점

셀렉터 안정성, 코드 패턴, 유지보수성을 평가합니다.

| 항목 | 감점 기준 | 감점 |
|------|-----------|------|
| `getByText()` 단독 사용 (exact 없이) | 1건당 | -2 |
| `getByText()` with `exact: true` (testid나 role로 대체 가능한 경우) | 1건당 | -1 |
| `getByRole('row').first()` 등 비특정 셀렉터 | 1건당 | -2 |
| regex 셀렉터가 데이터 포맷에 의존 (예: `/^전체 \d+$/`) | 1건당 | -1 |
| 하드코딩된 timeout 값 (상수 미사용) | 1건당 | -1 |
| `page.locator()` CSS 셀렉터 사용 (getByRole/testid 대체 가능 시) | 1건당 | -1 |
| 헬퍼 함수 미추출 (동일 패턴 3회 이상 반복) | 패턴당 | -2 |
| **testid 기반 셀렉터 사용** | 1건당 | **+1** (가점) |
| **getByRole 기반 셀렉터 사용** | 1건당 | **+1** (가점) |

#### 셀렉터 안정성 등급

| 등급 | 셀렉터 | 안정성 |
|------|--------|--------|
| A | `getByTestId('x')` | DOM 변경에 강함 |
| B | `getByRole('button', { name: 'x' })` | 접근성 기반, 안정적 |
| C | `getByPlaceholder('x')` | 입력 필드 한정 |
| D | `getByText('x')` | 텍스트 변경 시 깨짐 |
| F | `page.locator('.class-name')` | 스타일 변경 시 깨짐 |

### 2. 테스트 효과성 (Test Effectiveness) — 30점

각 TC가 실제로 버그를 잡을 수 있는지 평가합니다.

| 항목 | 감점 기준 | 감점 |
|------|-----------|------|
| **존재만 확인** — `toBeVisible()`만 있고 값/상태 미검증 | TC당 | -3 |
| **느슨한 검증** — `getByRole('row').first()` 등 아무 요소나 매칭 | TC당 | -2 |
| **단방향 검증** — 성공만 확인하고 실패 케이스 미검증 | TC당 | -1 |
| **상태 변화 미검증** — 액션 후 UI 변화를 확인하지 않음 | TC당 | -2 |
| **데이터 정합성 미검증** — API 응답 데이터와 UI 표시 값 비교 없음 | TC당 | -1 |
| **검증이 충분한 TC** — Given→When→Then 모두 의미 있는 검증 | TC당 | **+2** (가점) |

#### 검증 수준 등급

| 등급 | 설명 | 예시 |
|------|------|------|
| A | 값 + 상태 + 변화 검증 | 탭 클릭 → 활성 탭 변경 + 데이터 변경 + 카운트 일치 |
| B | 값 검증 | 요약 카드에 "180" 표시 확인 |
| C | 존재 검증 | `toBeVisible()` 만 |
| D | 무검증 | `test.skip` |

### 3. 시나리오 커버리지 (Scenario Coverage) — 30점

스펙 문서의 기능 대비 E2E가 얼마나 커버하는지 평가합니다.

| 항목 | 감점 기준 | 감점 |
|------|-----------|------|
| 스펙의 기본 흐름 미커버 | 흐름당 | -3 |
| 스펙의 대안 흐름 미커버 | 흐름당 | -2 |
| 스펙의 검증 조건 미커버 | 조건당 | -1 |
| 스펙의 @e2e auto TC 미구현 | TC당 | -2 |
| skip된 TC | TC당 | -1 |
| **스펙 범위 외 추가 TC** (엣지 케이스) | TC당 | **+1** (가점) |

## 실행 흐름

### Step 1: E2E 파일 분석

대상 파일을 읽고 다음을 추출합니다:
- TC 목록 (제목, 상태: pass/skip)
- 각 TC의 셀렉터 목록 + 등급 분류
- 각 TC의 expect 호출 목록 + 검증 수준 분류
- 헬퍼 함수 사용 여부
- 반복 패턴 감지

### Step 2: 대응 스펙 파일 찾기

E2E 파일명에서 스펙 파일을 역추적합니다:
- `e2e/purchase-request-detail.spec.ts` → `documents/specs/**/purchase-request-detail/**/*.md`
- 스펙의 기본 흐름, 대안 흐름, 검증 조건, @e2e TC 목록을 추출

### Step 3: 3가지 평가 실행

각 기준별로 점수를 산출합니다 (만점: 100점).

### Step 4: 리포트 생성

## 1. 스크립트 품질 (28/40)

### 셀렉터 분석

| TC | 셀렉터 | 등급 | 문제 |
|----|--------|------|------|
| TC-001 | `getByText('업체명(ID)')` | D | 라벨 텍스트 변경 시 깨짐. testid 권장 |
| TC-001 | `getByText('수량', { exact: true })` | D | 5개 요소와 부분 매칭. testid 권장 |
| TC-003 | `getByRole('row').first()` | D | 비특정 — 어떤 row인지 알 수 없음 |
| TC-002 | `getByRole('button', { name: /^전체 \d+$/ })` | B | 안정적이나 포맷 의존 |

### 개선 제안

1. 요약 카드에 `data-testid` 추가 → `getByTestId('summary-card-quantity')` 사용
2. 탭에 `data-testid` 추가 → `getByTestId('tab-all')` 사용
3. 반복되는 `page.goto(PAGE_URL) + waitForDetailPageReady` → `beforeEach`로 추출

## 2. 테스트 효과성 (15/30)

| TC | 검증 수준 | 문제 |
|----|-----------|------|
| TC-001 | C (존재만) | 카드 라벨만 확인, 값 미검증 ("정성균패션NEW0421", "180" 등) |
| TC-002 | B (값 검증) | 탭 존재 + 카운트 패턴 확인 |
| TC-003 | C (존재만) | row 존재만 확인, 탭 활성 상태/데이터 내용 미검증 |
| TC-005 | C (존재만) | 버튼 존재만 확인, enabled 상태 미검증 |

### 개선 제안

1. TC-001: 요약 카드 값 검증 추가 — `expect(page.getByTestId('company-name')).toContainText('정성균패션')`
2. TC-003: 탭 전환 후 데이터 검증 — 상태 배지 "승인" 표시 확인
3. TC-005: `toBeEnabled()` 추가
